# 技术文章写作框架与优化指南

根据用户提供的主题或文章路径，自动选择最佳写作框架并生成/优化技术文章。

## 〇、输入输出规范

### 输入格式

用户输入格式：`<主题> [要求]`

- **主题**（必填）：文章的核心主题，如 "Vue3 响应式原理"、"Docker 入门教程"
- **要求**（可选）：字数、风格、目标读者、重点内容等

**示例**：
- `Vue3 响应式原理，深度解析，3000字左右`
- `Docker 入门教程，面向后端开发者`
- `Redis vs Memcached 对比选型`

### 输出配置

| 配置项 | 默认值           | 说明 |
|-------|---------------|------|
| 输出目录 | `./src/news/` | 文章输出目录 |
| 文件格式 | Markdown      | `.md` 后缀 |
| 文件命名 | `{主题关键词}.md`  | 自动从主题提取关键词 |
| 语言 | 中文            | 根据用户语言自动适配 |

### 文件命名规则

1. 从主题中提取核心关键词（2-4个字）
2. 转换为小写英文或拼音
3. 用连字符连接
4. 示例：
   - "Vue3 响应式原理" → `vue3-reactive.md`
   - "Docker 入门教程" → `docker-guide.md`
   - "Redis vs Memcached 对比" → `redis-memcached-compare.md`

## 一、文章类型与框架选择

首先识别文章类型，选择对应的写作框架：

| 文章类型 | 核心目标 | 推荐框架 |
|---------|---------|---------|
| 入门科普 | 降低门槛，建立认知 | 入门科普型 |
| 实战教程 | 手把手教学，解决实际问题 | 实战教程型 |
| 原理解析 | 深入底层，理解本质 | 深度解析型 |
| 故障排查 | 定位问题，给出方案 | 问题排查型 |
| 技术选型 | 对比分析，辅助决策 | 对比选型型 |
| 经验分享 | 复盘总结，传递技巧 | 经验总结型 |

## 二、六种写作框架详解

### 框架1：入门科普型（5步走）

适用场景：编程语言入门、技术概念科普、工具基础介绍

```
1. 场景引入（100-200字）
   - 结合日常场景或痛点引出技术概念
   - 模板：「你是否遇到过...？这背后离不开...技术」

2. 核心定义（通俗解读）
   - 用直白语言解释，避免术语堆砌
   - 必须配一个简单类比

3. 核心价值（为什么需要）
   - 拆解解决的核心问题
   - 说明必要性和优势

4. 基础应用（简单上手）
   - 最基础的使用场景/操作步骤
   - 让读者知道"怎么用"

5. 总结与延伸
   - 提炼核心要点
   - 给出后续学习方向
```

### 框架2：实战教程型（6步走）

适用场景：编程实战、工具操作指南、脚本开发

```
1. 需求明确（开篇）
   - 清晰说明实现目标
   - 明确适用场景和读者

2. 前置准备
   - 环境、工具、依赖包
   - 必须标注版本要求

3. 分步操作（核心）
   - 按逻辑顺序拆解
   - 每步：操作内容 + 代码 + 说明
   - 关键步骤标注注意事项

4. 结果验证
   - 如何验证结果正确
   - 给出预期效果

5. 常见问题与避坑
   - 报错信息 + 原因 + 解决方法
   - 至少列出3个常见问题

6. 拓展优化（可选）
   - 进阶优化方向
```

### 框架3：深度解析型（5步走）

适用场景：技术原理解析、源码分析、算法原理

```
1. 引言（背景 + 核心问题）
   - 发展背景、应用场景
   - 提出要解析的核心问题

2. 核心原理拆解
   - 按逻辑分层拆解底层原理
   - 每个原理点讲清：是什么、怎么工作、为什么这么设计

3. 源码/案例佐证
   - 结合核心源码片段
   - 标注关键代码含义
   - 或用实际案例验证

4. 优缺点/适用场景分析
   - 客观分析优势和局限性
   - 明确适用和不适用场景

5. 总结与展望
   - 提炼核心原理要点
   - 展望未来优化方向
```

### 框架4：问题排查型（5步走）

适用场景：程序bug排查、服务器故障、环境配置错误

```
1. 问题描述（精准具体）
   - 问题现象、出现场景
   - 完整报错信息
   - 环境版本信息

2. 问题定位（排查步骤）
   - 从易到难拆解
   - 每步：排查方法 + 排查结果

3. 原因分析
   - 核心原因
   - 结合技术原理说明

4. 解决方案（可落地）
   - 具体解决步骤
   - 代码示例
   - 可补充多种方案

5. 预防措施
   - 避免复现的方法
   - 开发规范建议
```

### 框架5：对比选型型（5步走）

适用场景：技术选型、工具对比、编程语言对比

```
1. 选型背景（需求引入）
   - 选型场景和核心需求
   - 明确核心指标

2. 对比对象与指标确定
   - 对比的技术/工具及版本
   - 确定核心对比指标

3. 详细对比（核心）
   - 按指标逐一对比
   - 使用表格呈现
   - 客观呈现优缺点

4. 选型建议（场景适配）
   - 结合不同场景给建议
   - 让读者能对应自身需求

5. 总结
   - 提炼核心对比结论
   - 强调"没有最好，只有最适合"
```

### 框架6：经验总结型（5步走）

适用场景：项目实战复盘、技术经验分享、踩坑总结

```
1. 实战背景（开篇）
   - 项目背景、核心需求
   - 遇到的核心挑战

2. 核心问题与解决方案
   - 分点说明问题
   - 重点分享思考过程

3. 经验提炼（可复用技巧）
   - 总结可复用技巧
   - 优化思路

4. 踩坑教训（避坑指南）
   - 分享错误和踩坑
   - 说明原因及改进方法

5. 总结与反思
   - 收获和不足
   - 后续改进方向
```

## 三、通用写作技巧

### 开篇技巧（3秒抓住注意力）

| 方法 | 模板 | 示例 |
|-----|------|-----|
| 痛点切入 | 「写文章总被说...？其实你只是...」 | 「写代码总被review打回？其实是你的命名不规范」 |
| 疑问开头 | 「同样是...，为什么有的人...？」 | 「同样是写接口，为什么有的人代码更优雅？」 |
| 场景代入 | 「你有没有过这样的经历：...」 | 「你有没有遇到过：本地能跑，部署就报错？」 |
| 价值前置 | 「读完这篇文章，你能掌握...」 | 「读完本文，你能独立完成一个爬虫项目」 |

### 中间内容技巧

1. **分点清晰，格式统一**
   - 用「数字 + 关键词」开头
   - 同类型内容保持格式一致

2. **观点 + 案例结合**
   - 每个观点配具体案例
   - 案例要具体、简短

3. **善用过渡句**
   - 段落之间用一句话过渡
   - 「掌握了...，接下来我们看看...」

### 结尾技巧

| 方法 | 适用场景 | 模板 |
|-----|---------|-----|
| 总结升华 | 所有类型 | 「以上X种...，无需全部掌握，选1-2种反复练习...」 |
| 行动引导 | 教程类 | 「今天就试着用...，完成一个...」 |
| 延伸思考 | 深度类 | 「除了以上...，你平时还常用哪种...？」 |

### 细节技巧

1. **简洁为主**：删除多余修饰词，多用短句
2. **用词精准**：用具体数据替代模糊词（「超过80%」而非「很多人」）
3. **避免口语化**：根据场景调整语气，避免网络流行语
4. **反复检查**：逻辑连贯、无错别字、语句通顺

## 四、通用文章结构（灵活组合）

| 结构 | 核心逻辑 | 适用场景 |
|-----|---------|---------|
| 总分总 | 开篇点题 → 分点展开 → 总结升华 | 万能适配，新手首选 |
| 问题-分析-解决 | 提出问题 → 分析问题 → 解决方案 | 干货教程、观点文 |
| 并列式 | 多维度并列展开 | 清单文、测评文 |
| 递进式 | 从浅到深层层递进 | 深度评论、散文 |
| 对比式 | 正反/新旧/优劣对比 | 测评文、观点文 |

## 五、执行流程

当用户提供主题或文章路径时：

### 步骤1：需求分析
- 如果是主题：直接进入框架选择
- 如果是文件路径：读取文件，诊断现有问题
- 解析用户要求：字数、风格、目标读者、重点内容

### 步骤2：框架选择
根据主题/内容自动匹配最佳框架（参考"六、快速参考"）

### 步骤3：内容生成/优化

**生成新文章**：
1. 按选定框架结构填充内容
2. 应用开篇、中间、结尾技巧
3. 确保观点有案例支撑
4. 检查格式统一性
5. 添加 Markdown frontmatter（title, date, tags）

**优化现有文章**：
1. 检查结构完整性，补全缺失部分
2. 优化开篇（是否够吸引？）
3. 丰富中间内容（观点是否有案例？）
4. 强化结尾（是否有行动引导？）
5. 细节润色（数据是否具体？表述是否清晰？）

### 步骤4：文件写入

1. **确定输出路径**
   - 默认目录：`./src/news/`
   - 用户指定目录：优先使用用户指定的路径
   - 如果目录不存在：自动创建

2. **生成文件名**
   - 从主题提取关键词
   - 按命名规则生成文件名
   - 如有重名：添加序号后缀（如 `docker-guide-2.md`）

3. **写入文件**
   - 使用 Write 工具写入完整文章
   - 包含 frontmatter 和正文内容

4. **输出报告**

```report
### ✅ 文章生成完成

**📄 文件**：`{输出路径/文件名}`
**🎯 框架**：{框架名称}
**📝 字数**：{实际字数}

**📌 文章结构**：
1. {章节1标题}
2. {章节2标题}
3. ...

**💡 核心亮点**：
- {亮点1}
- {亮点2}
```

## 六、快速参考

### 一句话判断用哪个框架

- 「我想教读者从零学会某个技术」→ **入门科普型**
- 「我想手把手教读者完成某个操作」→ **实战教程型**
- 「我想深入讲解某个技术原理」→ **深度解析型**
- 「我解决了一个bug想分享」→ **问题排查型**
- 「我想对比几个技术帮读者选型」→ **对比选型型**
- 「我想分享项目实战经验」→ **经验总结型**

### 必须遵守的原则

1. **开头必须吸引**：前3句话抓住读者
2. **观点必有案例**：每个观点配具体示例
3. **数据必须具体**：用精确数字替代模糊词
4. **结构必须清晰**：分点编号，格式统一
5. **结尾必给价值**：总结或行动引导
